## Introduction

The Bethlemitas platform is built on **Laravel 11**, following the Model-View-Controller (MVC) architectural pattern. This SaaS application manages student psycho-orientation services with a focus on role-based access control and educational workflows.

## MVC Architecture

### Model Layer

Eloquent ORM models represent database tables and handle:
- Data relationships (students, teachers, referrals)
- Business logic (scopes, accessors, mutators)
- Database interactions

**Key Models**: Users_teacher, Users_student, Referral, Psychoorientation, Group, Degree, Area

### Controller Layer

Controllers handle HTTP requests and coordinate application flow:

- **AuthController**: Authentication and password reset
- **PsicoController**: Psychologist operations (reports, student management)
- **CreateReferralController**: Student referral creation and management
- **CreateController**: User (teacher/psychologist) CRUD operations
- **ProfileController**: User profile management

### View Layer

Blade templates render the user interface, organized by role:
- `auth/`: Login and password reset
- `teacher/`: Teacher-specific views
- `psycho/`: Psychologist-specific views
- `academic/`: Coordinator administrative views
- `layout/`: Shared layouts and components

## Role-Based Access Control

The platform implements three primary roles using Spatie Laravel Permission:

<Note>
The application uses `hasRole()` checks throughout controllers and middleware to enforce authorization.
</Note>

### User Roles

| Role | Description | Key Responsibilities |
|------|-------------|---------------------|
| **coordinador** | Academic Coordinator | User management, system configuration |
| **docente** | Teacher | Student referrals, class management |
| **psicoorientador** | Psychologist | Student evaluation, reports, PIAR/DUA |
| **estudiante** | Student | Passive role for student records |

### Route Protection Example

```php
// Routes protected by RoleMiddleware (coordinador only)
Route::middleware([RoleMiddleware::class])->group(function () {
    Route::get('/create/user', [CreateController::class, 'create_user'])
        ->name('create.user');
    Route::post('/store/user', [CreateController::class, 'store_user'])
        ->name('store.user');
    Route::get('/listing/users', [CreateController::class, 'index_users'])
        ->name('index.users');
});

// Routes for both docente and psicoorientador
Route::middleware([RolePsicoorientadorAndDocenteMiddleware::class])->group(function () {
    Route::get('/create/referral', [CreateReferralController::class, 'create_referral'])
        ->name('create.referral');
    Route::post('/store/referral', [CreateReferralController::class, 'store_referral'])
        ->name('store.referral');
});
```

## Database Architecture

### Core Tables

- **users_teachers**: Teacher and psychologist accounts
- **users_students**: Student records
- **referrals**: Student referrals from teachers
- **psychoorientations**: Psychological evaluation reports
- **groups**: Academic groups/classes
- **degrees**: Grade levels
- **areas**: Subject areas
- **states**: Student status (activo, en PIAR, en DUA)

### Pivot Tables

- **teachers_areas_groups**: Links teachers to subjects and groups
- **users_load_degrees**: Assigns psychologists to grade levels
- **users_load_groups**: Assigns teachers to groups
- **users_load_areas**: Assigns teachers to subject areas

## Application Flow

### Student Referral Workflow

1. **Teacher creates referral** → `CreateReferralController@store_referral`
2. **Student record created** with state "activo"
3. **Email notification** sent to assigned psychologist
4. **Psychologist reviews** → `PsicoController@detailsReferral`
5. **Report created** → `PsicoController@store_report_student`
6. **State updated** to PIAR/DUA if needed

```php
        $user->save();

        $referral->id_user_teacher = $id_teacher;
```

## Key Design Patterns

### Authentication Guard

The platform uses a single authentication guard for `users_teachers` table with custom middleware:

```php
// Only active users (id_state = 1) can log in
if (Auth::attempt([
    'number_documment' => $request->number_documment,
    'password' => $request->password,
    'id_state' => 1
])) {
    return redirect()->route('dashboard');
}
```

See app/Http/Controllers/auth/AuthController.php:35-42

### Scope Queries

Models use query scopes for reusable filters:

```php
// In Users_teacher model
public function scopeActivos($query)
{
    return $query->where('id_state', 1);
}

public function scopePiar($query)
{
    return $query->where('id_state', 2);
}

// Usage:
$activeTeachers = Users_teacher::activos()->get();
```

Controllers use eager loading to prevent N+1 queries:


## Security Features

<Warning>
All routes except authentication are protected with the `auth` middleware and role-specific middleware.
</Warning>

### Protection Layers

1. **Authentication**: `Authenticate` middleware checks user login status
2. **State Verification**: Suspended users (id_state = 2) are automatically logged out
3. **Role Authorization**: Role-specific middleware prevents unauthorized access
4. **Cache Prevention**: `PreventBackHistoryMiddleware` prevents browser back-button issues
5. **CSRF Protection**: Laravel's built-in CSRF protection on all forms

### Middleware Stack

```php
Route::middleware([PreventBackHistoryMiddleware::class])->group(function () {
    Route::middleware(['auth'])->group(function () {
        // All protected routes
    });
});
```
